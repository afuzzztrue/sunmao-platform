# 榫卯非遗文化传承平台 - 前后端模块对应解析

> 本文档详细解析后端 `src` 下每个模块的功能，并建立与前端微信小程序各模块的对应关系。

---

## 特别说明：关于分层架构与阿里Java开发手册的适配

本项目为**微信小程序后端**，采用简化分层架构（Controller → Service → Mapper → Entity）。

### 与阿里巴巴Java开发手册的关系

阿里巴巴Java开发手册（v1.7.1）推荐的分层领域模型规约：
- **DO** (Data Object)：与数据库表结构一一对应
- **DTO** (Data Transfer Object)：Service层向外传输的对象
- **VO** (View Object)：Controller层返回给前端的展示对象
- **BO** (Business Object)：Service层输出的封装业务逻辑的对象

### 本项目的实际选择

考虑到微信小程序后端的**项目规模和开发效率**，本项目采用以下简化方案：

| 手册推荐 | 本项目实际 | 说明 |
|----------|-----------|------|
| `xxxDO.java` | `User.java`、`Article.java` 等 `Entity` | 命名风格差异，`Entity` 也是业界通用命名 |
| Controller 返回 `VO` | Controller 返回 `Map<String, Object>` 或 `Result<Entity>` | 通过 Map 手动封装响应数据，已过滤敏感字段 |
| Service 层使用 `DTO` | Service 层直接传递 `Entity` | 简化对象转换，适合中小型项目 |

### 安全处理措施

虽然未严格使用 `VO` 对象，但本项目在 **Controller 层** 已做敏感字段过滤：

```java
// UserController.login() 示例
@PostMapping("/login")
public Result<Map<String, Object>> login(@RequestParam String account, @RequestParam String password) {
    User user = userService.login(account, password);
    
    // 手动封装返回数据，过滤敏感字段
    Map<String, Object> result = new HashMap<>();
    result.put("userId", user.getUserId());
    result.put("nickname", user.getNickname());
    result.put("avatar", user.getAvatar());
    result.put("studyHours", user.getStudyHours());
    // 注意：password、openid、unionid 等敏感字段不返回
    
    return Result.success(result);
}
```

### 后续扩展建议

如项目后续扩展（如增加**管理后台**、**开放第三方接口**），建议按阿里规约添加 `VO/DTO` 分层：

```
当前：Controller → Service → Mapper → Entity
扩展：Controller → Service → DTO/BO → Mapper → Entity
              ↓
            VO（返回前端）
```

> **总结**：本项目当前架构符合微信小程序后端的常见实践，在开发效率和代码简洁性之间取得了平衡。如需更严格的分层规范，可参考阿里巴巴Java开发手册进行扩展。

## 二、后端模块逐层解析

### 1. common 包 - 通用工具类

| 文件 | 功能说明 | 对应前端模块 |
|------|----------|-------------|
| `Result.java` | 统一响应结果封装，所有接口返回格式为 `{code, msg, data}` | **所有页面** - 前端解析后端响应的基础结构 |
| `PageResult.java` | 分页响应封装，包含 `total/list/current/size/pages` | **首页/分类页/动态页** - 列表分页加载 |
| `BusinessException.java` | 业务异常类，用于抛出业务逻辑错误（如"用户不存在"） | **登录页/注册页** - 显示错误提示 |
| `GlobalExceptionHandler.java` | 全局异常处理器，统一捕获异常并包装为 Result 返回 | **所有页面** - 保证前端收到标准错误格式 |

**核心作用**：保证前后端通信的数据格式统一，前端只需解析 `Result` 结构即可处理所有接口响应。

| `CourseServiceImpl.java` | `recordDownload`：记录课程下载，已下载则跳过 | **分类页-教程**：下载课程 |

## 四、数据流示意图

> 依据阿里巴巴Java开发手册（泰山版/嵩山版）及Spring Boot分层架构规范：
> - **强制**：遵循 Controller → Service → Mapper 的单向调用链
> - **强制**：严禁 Service 循环依赖
> - **规范**：Entity 作为纯数据载体，在各层间传递，映射数据库表结构
> - **规范**：Controller 层只做参数校验和结果封装，不处理业务逻辑
> - **规范**：Service 层处理业务逻辑和事务边界
> - **规范**：Mapper 层只做数据持久化，不掺杂业务逻辑

### 简化版数据流

```
前端页面 → wx.request → Controller → Service → Mapper → MySQL数据库
              ↑___________________________________________|
                        Result {code, msg, data}
```

### 各层职责与数据流转（阿里规约）

| 层级 | 核心职责 | 能做什么 | 严禁做什么 | Entity 的作用 |
|------|----------|----------|-----------|--------------|
| **Controller** | 接收请求、参数校验、调用Service、返回响应 | 参数校验、调用Service、封装Result | **严禁处理业务逻辑** | 接收前端JSON → 转为Entity；将Entity封装到Result |
| **Service** | 业务逻辑封装、事务管理 | 业务规则处理、事务控制、调用Mapper | **严禁直接操作数据库** | 在各方法间传递业务数据；对Entity进行业务加工 |
| **Mapper** | 数据持久化、SQL执行 | 执行CRUD SQL、结果映射 | **严禁处理业务逻辑** | 将Entity映射为INSERT/UPDATE；将查询结果映射为Entity |
| **common** | 统一响应、异常处理 | 封装标准格式、全局异常捕获 | — | 与Entity配合封装响应数据 |

### 单向调用链（强制）

```
HTTP请求
    ↓
Controller（参数校验）
    ↓ 调用 Service 接口
Service（业务逻辑 + 事务）
    ↓ 调用 Mapper 接口
Mapper（SQL执行）
    ↓ JDBC
MySQL（数据存储）
    ↑
Mapper（返回Entity）
    ↑
Service（业务加工）
    ↑
Controller（封装Result）
    ↑
HTTP响应（JSON）
```

### 本项目中的典型调用示例

以 **用户登录** 为例：

```
前端（login.js）
    ↓ POST /api/user/login
UserController.login(account, password)
    ↓ 参数校验（非空校验）
    ↓ 调用 userService.login(account, password)
UserServiceImpl.login(account, password)
    ↓ 业务逻辑：判断账号类型（邮箱/手机号）
    ↓ 调用 userMapper.selectByPhone() 或 selectByEmail()

    ↓ 执行 SQL 查询
MySQL.user 表
    ↑ 返回 User Entity
UserMapper
    ↑ 返回 User Entity
UserServiceImpl
    ↑ 校验密码（MD5加密比对）
    ↑ 校验用户状态
    ↑ 返回 User Entity
UserController
    ↑ 将 User Entity 封装为 Map（脱敏处理）
    ↑ 封装为 Result.success(data)

    ↑ 解析 Result，提取用户信息，保存到本地存储
```

## 五、关键技术点总结

### 1. 统一响应格式

所有接口返回统一的 `Result` 结构：
```json
{
  "code": 200,      // 200成功，其他为错误码
  "msg": "success", // 提示信息
  "data": {}        // 业务数据
}
```

前端只需判断 `code === 200` 即可知道请求是否成功。

### 2. 分页查询

列表接口支持分页参数 `current` 和 `size`，返回 `PageResult`：
```json
{
  "code": 200,
  "data": {
    "total": 100,
    "list": [...],
    "current": 1,
    "size": 10,
    "pages": 10
  }
}
```

### 3. 切换型操作（Toggle）

点赞、收藏、关注均采用"切换"模式：
- 已存在记录 → 删除记录（取消操作）
- 不存在记录 → 插入记录（执行操作）
- 同步更新目标表的计数字段

### 4. 登录方式

支持三种登录方式：
- **微信登录**：传入 `code`，后端调用微信接口获取 `openid`，自动注册/更新用户
- **手机号登录**：传入手机号和密码
- **邮箱登录**：传入邮箱和密码

### 5. 密码加密

使用 MD5 加密存储（`DigestUtils.md5DigestAsHex`），实际生产环境建议使用 BCrypt。
