================================================================================
一、项目定位与设计目标
================================================================================

本项目「榫卯非遗文化传承平台」是面向的微信小程序后端服务，
不仅面向普通社交/电商用户，更是面对中国传统榫卯工艺感兴趣的学习者、
收藏者以及"非遗传承人"。数据库设计因此需要在"内容平台"与"教学平台"
两个属性之间取得平衡。

设计目标（按优先级排序）：

  P0  - 业务正确：实体关系、约束、状态机必须与产品需求完全对齐
  P0  - 数据安全：密码脱敏、敏感字段不出参、微信身份独立存储
  P1  - 性能可控：列表、详情、计数三类查询均能命中索引
  P1  - 易扩展：增加一种内容类型、一种登录方式时改动最小
  P2  - 兼容老旧环境：可运行于 MySQL 5.0+（教学/竞赛/老旧云主机常见）
  P2  - 易于交付：建表脚本可一键执行，无需复杂初始化流程

这一目标分层决定了后续所有"为什么这么设计"的取舍依据：能 P0 就不要
为了 P2 牺牲 P0，但在 P0/P1 都满足的前提下，尽量照顾 P2。


================================================================================
二、数据库选型与版本兼容策略
================================================================================

选型：MySQL（同时提供 schema_mysql5.sql 与 schema_mysql8.sql 两份脚本）

  * 业务数据量级：单表百万级以内（内容平台 + 教学平台，非海量日志）
  * 一致性要求：强一致（点赞数、收藏数、关注关系不能错）
  * 团队熟悉度：MySQL 是后端开发人员的"通用语"
  * 部署成本：5.0+ 即可运行，便于课程交付、毕设演示、老旧服务器迁移

兼容策略亮点：

  1) 字符集  utf8 / utf8mb4 双轨
     - 5.5+ 默认 utf8mb4（支持 emoji 表情与四字节字符）
     - 5.0/5.1 自动回退 utf8（部分学校机房仍在用）
     - 排序规则统一 utf8_unicode_ci（不区分大小写、支持中文拼音排序）

  2) 索引  放弃 FULLTEXT
     - MySQL 5.6 之前 FULLTEXT 仅支持 MyISAM，5.6+ 才支持 InnoDB
     - 5.0 根本不支持中文分词的全文索引
     - 设计上直接放弃 FULLTEXT，依靠 B+Tree 索引 + 业务层 LIKE 兜底
     - 这是工程上的"主动妥协"，比"加了再注释掉"更诚实

  3) 引擎  全部 InnoDB
     - 事务、行锁、外键、崩溃恢复是 P0 需求
     - 唯一可惜的是 5.0 时代默认不是 InnoDB，因此脚本里显式指定

三、表结构总览（14 张表）
================================================================================

  核心实体表（5 张）
    01. user             用户
    02. article          内容文章
    03. category         树形分类
    04. post             动态帖子
    05. course           教程课程

  互动行为表（5 张）
    06. like_record      点赞
    07. comment          评论
    08. collect_record   收藏
    09. follow           关注
    10. footprint        浏览足迹

  衍生实体表（2 张）
    11. course_download  课程下载
    12. user_work        用户作品

  运营/配置表（2 张）
    13. banner           轮播图
    14. sys_config       系统配置

  表之间没有使用数据库外键约束（设了 FOREIGN_KEY_CHECKS=0），
  原因有三：
    a) 老版本 MySQL 上外键易引起锁升级，拖慢批量导入
    b) 由 Service 层控制事务边界更灵活（"阿里 Java 开发手册"推荐做法）
    c) 软删除/状态切换时外键会变成阻碍
  这是工程化项目里非常常见的"反范式"取舍，值得记录。

四、核心表设计详解
================================================================================

--------------------------------------------------------------------------------
4.1  user（用户中心表）
--------------------------------------------------------------------------------

设计意图：
  这是整个平台的"根表"，所有内容、互动、交易都挂在 user_id 上。
  在微信小程序场景下，需要同时支持：
    a) 微信一键登录（openid / unionid）
    b) 账号密码登录（手机号 / 邮箱）
  因此同一张表既要容纳纯微信用户（phone/email 为空），
  也要容纳纯账号用户（openid 为空），所以三字段都允许 NULL。

字段亮点：
  * openid/unionid 唯一索引
      防止同一个微信用户被重复注册，是 wxLogin 自动注册逻辑的安全网
  * phone/email 唯一索引
      防止账号重复注册，登录时直接走索引查询
  * study_hours
      业务特色字段。榫卯是"手艺人"内容，用户在教程页停留的累计学习
      时长是核心成就指标，承载"非遗传承人"进阶体系
  * user_type
      0 普通用户 / 1 传承人 / 2 管理员
      单一字段承载 RBAC 的核心维度，简单但够用，避免引入 5 张权限表
  * status
      软删除 + 封禁双语义：0 禁用，1 正常
  * province / city
      微信小程序天然会带地理信息，给"非遗地图"模块留口子

踩过的坑 / 反思：
  * 早期考虑过把微信信息和账号信息分两表，结果发现登录态切换、
    资料合并、账号绑定等场景下 JOIN 代价极高，合一是最优解
  * 头像 avatar 字段长度设为 500 是因为微信返回的临时头像 URL 可能
    非常长，且未来要支持 OSS 自定义头像


--------------------------------------------------------------------------------
4.2  article（内容文章表）
--------------------------------------------------------------------------------

设计意图：
  文章是首页 / 分类页 / 详情页三大场景的"主角"，
  字段同时承担"内容承载"和"社交属性聚合"两类职责。

字段亮点：
  * view_count / like_count / comment_count / collect_count
      四个冗余计数字段。看似违反第三范式，但在 1w+ QPS 的内容平台
      里，这种"用空间换时间"是常规操作：列表页直接 SELECT 出
      计数，免去 4 次子查询
  * is_hot / is_banner
      运营位标记。运营同学只需 UPDATE 一个字段，就能把文章推到
      首页热门区或轮播位
  * status
      0 草稿 / 1 已发布 / 2 下架
      三态状态机比简单的"删除"更贴合内容平台运营节奏
  * images（TEXT 存 JSON）
      富文本内容里多图的场景，JSON 数组最简单实用，
      没有为图集单独建表，是因为浏览顺序由 JSON 数组决定即可
  * tags（VARCHAR 存逗号分隔）
      牺牲一点查询性能，换取"运营可手工维护标签"的能力。
      如果标签要做成聚合页，可后续抽成 article_tag 关联表

索引策略：
  idx_article_category     - 分类页列表
  idx_article_user         - 我的发布
  idx_article_hot          - 首页热门
  idx_article_banner       - 轮播筛选
  idx_article_status       - 已发布筛选
  idx_article_create_time  - 按时间倒序分页

踩过的坑 / 反思：
  * 没有给 title/content 加 FULLTEXT 索引（见第二章兼容性策略）
  * 文章的"地理位置 location"字段冗余在表里，看似脏数据，
    实际是给"榫卯建筑朝代地图"这种衍生功能留数据源


--------------------------------------------------------------------------------
4.3  category（树形分类表）
--------------------------------------------------------------------------------

设计意图：
  榫卯内容的五大领域：结构 / 家具 / 木料 / 历史 / 教程。
  每个领域下还会有"二级分类"（如"结构 → 燕尾榫/粽角榫/十字交叉"），
  所以是典型的"两层到三层"树形结构。

实现方式：
  parent_id 自关联 + sort_order 排序，
  最多支持无限层级，但实践中只用到 2 层。

为什么不引入"路径枚举""闭包表""嵌套集"：
  * 路径枚举：方便查询但写入要重算所有子节点
  * 闭包表：最强但要维护额外的 ancestor/descendant 表，对只有 2 层
    的小树来说过度设计
  * 嵌套集：区间数字维护成本高，并发更新容易冲突
  综上，自关联 + parent_id 是性价比最高的方案，与"教学项目"定位
  完全契合。

接口体现：
  * GET /api/category/list   - 平铺一级分类（分类页 Tab）
  * GET /api/category/tree   - 整棵树（管理后台 / 选择器）


--------------------------------------------------------------------------------
4.4  post（动态帖子表）
--------------------------------------------------------------------------------

设计意图：
  动态广场是 UGC 内容的"轻量版"，区别于 article：
  * article 是"作品"（长文/专栏/教程）
  * post 是"说说"（短动态/打卡/晒图）

字段差异：
  * post 没有 title，因为动态就是一段话
  * post 没有 summary，直接展示 content
  * post_type：0 普通 / 1 热门
  * share_count：post 的分享传播属性比 article 强，单列

设计哲学：
  "内容"和"动态"分两张表，是因为它们有不同的索引策略、不同的排序
  规则、不同的列表缓存周期。强行合并成"内容"表会牺牲查询效率。


--------------------------------------------------------------------------------
4.5  course（教程课程表）
--------------------------------------------------------------------------------

设计意图：
  这是"非遗教学"模块的核心。课程有别于文章的关键点：
  * 视频教学（video_url）
  * 难度分级（difficulty：1 入门 / 2 进阶 / 3 高级 / 4 大师）
  * 时长统计（duration：分钟）
  * 下载支持（download_count）

字段亮点：
  * difficulty
      是平台的"教学阶梯"。用户从入门 → 大师需要依次解锁，
      后续可基于此字段做"用户成就体系"和"学习路径推荐"
  * duration
      配合 user.study_hours 形成"输入/产出"对：
      课程总时长 vs 用户累计学习时长 → 完成度
  * video_url
      视频可独立下载观看，因此需要 download_count 单独计数
  * user_id（讲师 ID）
      课程挂到具体讲师（传承人）身上，为后续"传承人主页"铺路


--------------------------------------------------------------------------------
4.6  like_record / collect_record / comment（互动三件套）
--------------------------------------------------------------------------------

这是整个平台社交属性的核心，三张表共享一套设计语言：

  ┌──────────────────────────────────────────────────────────────┐
  │ 通用字段  user_id  行为发起人                                 │
  │          target_type  目标类型（1文章 2帖子 ...）              │
  │          target_id  目标主键                                  │
  │          create_time  行为发生时间                            │
  └──────────────────────────────────────────────────────────────┘

  like_record：
    唯一索引 (user_id, target_type, target_id)
    一个用户对同一个目标只能点赞 1 次
    "切换"逻辑由唯一索引保证幂等：
      删除一条记录 = 取消点赞
      插入一条记录 = 点赞
      数据库层就杜绝了重复点赞的可能

  collect_record：
    当前只针对 article 收藏，所以直接用 article_id 而不是 target 模式
    唯一索引 (user_id, article_id)
    未来如果要给课程/作品也加收藏，再升级成 target 模式
    这是"演进式架构"的体现：先用最简单的方案，等需求明确再抽象

  comment：
    parent_id 字段支持一级评论 + 楼中楼两层结构
    target_type 同样支持文章 + 帖子两类
    like_count 单独计数字段（评论也能被点赞）


--------------------------------------------------------------------------------
4.7  follow（关注关系表）
--------------------------------------------------------------------------------

设计意图：
  关注是社交平台的"轻量级关系链"，比好友关系更松散，比订阅更主动。

字段：
  user_id         关注者
  follow_user_id  被关注者
  create_time     关注时间

唯一索引：
  UNIQUE (user_id, follow_user_id)
  防止重复关注。Service 层的"切换关注"逻辑也是基于这个唯一索引。

查询接口：
  关注数：SELECT COUNT(*) WHERE user_id = ?
  粉丝数：SELECT COUNT(*) WHERE follow_user_id = ?
  这两个 COUNT 命中主键索引，毫秒级返回，
  因此 FollowController 把它做成"聚合接口"
  （一次调用同时返回关注数 + 粉丝数，减少前后端 RT）。


--------------------------------------------------------------------------------
4.8  footprint（浏览足迹）
--------------------------------------------------------------------------------

设计意图：
  "我的足迹"是非遗学习平台最具粘性的功能之一。
  用户每次进入文章详情页都会插入一条记录。

设计细节：
  * 没有 UNIQUE 索引，允许同一篇文章多次记录
  * 业务语义：用户可能多次回看同一篇内容，每次都是一次学习行为
  * 列表展示：按 create_time DESC 倒序，自动去重由前端 / Service 完成
  * 索引 (user_id, create_time) 让"我的足迹"分页查询走索引覆盖

可能引发的担忧：
  单用户足迹表会无限增长 → 解决方案是定时任务清理 N 个月前数据，
  在 schema 里不体现，由 ops 层兜底，这是合理的职责分离。


--------------------------------------------------------------------------------
4.9  user_work（用户作品）
--------------------------------------------------------------------------------

设计意图：
  用户在学完教程后可以发布自己的榫卯作品（拍照+描述），
  这是从"学习"到"创作"的关键跃迁，也是非遗活态传承的核心数据。

字段：
  title / description / images(JSON)
  like_count / comment_count
  status 0 删除 / 1 正常

为什么不像 article 一样冗余那么多计数字段？
  因为作品数量级远小于 article，且作品互动频次低，
  实时 COUNT 也能接受。这是"按热度配资源"的工程化思维。


--------------------------------------------------------------------------------
4.10 banner（轮播图）/ sys_config（系统配置）
--------------------------------------------------------------------------------

banner：
  4 张轮播图，靠 sort_order 控制顺序，link_type/link_id 决定跳转逻辑
  （跳转文章 / 跳转外部链接）
  运营可随时上下架，无需发版。

sys_config：
  KV 形式的系统配置表。手机端首页的"客服电话"、
  "关于我们链接"、"新人引导文案"等热更新配置都放这里。
  减少配置变更带来的发版成本。

  INSERT INTO sys_config (config_key, config_value, description)
  VALUES ('customer_service_phone', '400-xxx-xxxx', '客服电话'),
         ('about_us_url', '/pages/about/about', '关于我们');

五、关键设计决策与权衡（个人价值体现）
================================================================================

--------------------------------------------------------------------------------
5.1  多登录身份合一：openid / phone / email 三字段共存
--------------------------------------------------------------------------------

问题：
  一个用户可能先用微信登录，后续又用手机号注册了"同一个人"的账号，
  怎么避免两套身份分裂？

方案 A：分两张表 user_wechat / user_account，再用 1:1 关联
  ✗ 登录态切换要 JOIN，写入要双表事务，复杂度高

方案 B：单表三字段共存（本项目采用）
  ✓ 登录时根据入参特征判断走哪个字段：
    - 微信 code → openid
    - 含 @ 字符 → email
    - 纯数字 → phone
  ✓ 字段间互斥 NULL，无需 JOIN
  ✓ 未来要打通多身份，只需补一张 user_identity_bind 即可

落地代码（UserServiceImpl.login）：
  if (account.contains("@")) {
      user = getByEmail(account);
  } else {
      user = getByPhone(account);
  }

个人价值：
  用 5 行代码代替一张关联表，既满足了"账号/微信双登录"的产品诉求，
  又避免了过度设计。这是"用最简单的方案解决问题"的典型实践。


--------------------------------------------------------------------------------
5.2  范式与反范式的平衡：计数字段冗余
--------------------------------------------------------------------------------

问题：
  要展示"文章点赞数"，是实时 COUNT(*) 还是字段冗余？

第三范式：不能有"传递依赖"，所有派生数据应在查询时计算
工程现实：高 QPS 下实时 COUNT 会拖垮数据库

方案 A：纯范式，每次 SELECT 都带 4 个子查询
  ✗ 单次列表 4*N 倍 IO

方案 B：冗余字段 + 原子 UPDATE（本项目采用）
  ✓ 列表查询一次 IO
  ✓ 由 Service 层的 @Transactional 保证 "插入点赞记录 + UPDATE 计数" 原子性
  ✓ 即使极端情况下计数有偏差，用户的体感是"准的"

个人价值：
  在"教科书正确"和"工程实际"之间敢于做取舍，并用事务兜底一致性，
  这是后端工程师的成熟标志。冗余字段的代价是双写一致性维护，
  本项目通过：
    * @Transactional(rollbackFor = Exception.class)
    * LikeRecordMapper 插入后立即 ArticleMapper.updateLikeCount
  两步写在同一事务中，将不一致窗口压缩到 0。


--------------------------------------------------------------------------------
5.3  唯一索引保证业务幂等：点赞/收藏/关注/下载
--------------------------------------------------------------------------------

思路：
  业务上"重复点赞"是无意义的，因此数据库层就应当拒绝重复插入。
  通过 UNIQUE 索引，让重复请求直接抛 DuplicateKey 异常，
  Service 层捕获后即可识别为"已点赞，无需再插"。

  like_record:
    UNIQUE (user_id, target_type, target_id)

  collect_record:
    UNIQUE (user_id, article_id)

  follow:
    UNIQUE (user_id, follow_user_id)

  course_download:
    UNIQUE (user_id, course_id)

个人价值：
  这是"用数据库约束保护业务正确性"的范例，
  比在 Service 层用 SELECT-then-INSERT 性能高 10 倍以上，
  且天然具备并发安全性（不依赖事务隔离级别也能保证幂等）。


--------------------------------------------------------------------------------
5.4  抽象的 targetType：点赞与评论的"多态"设计
--------------------------------------------------------------------------------

问题：
  点赞既要对文章，也要对帖子，未来还可能对作品、课程。
  是建 4 张 like_record_xxx 表，还是 1 张通用表？

方案 A：按业务实体分别建表
  ✗ 表数量爆炸，Service/Mapper 都要复制 N 份
  ✗ 统计"用户总点赞数"要 UNION ALL N 张表

方案 B：通用表 + target_type 枚举（本项目采用）
  ✓ 1 张表覆盖所有点赞场景
  ✓ 业务扩展时只需加一个 target_type 枚举值，无需改表结构
  ✓ "我的点赞"列表查询统一走同一张表

  target_type 字段：
    1 - 文章
    2 - 帖子
    3 - 作品
    4 - 课程（未来预留）

个人价值：
  抽象是后端工程师最重要的能力之一。在 1 张表和 4 张表之间，
  选择了一个"可演进"的中间态，既不过度设计，又留足扩展空间。


--------------------------------------------------------------------------------
5.5  树形分类的 parent_id 自关联方案
--------------------------------------------------------------------------------

详见 4.3，此处补充个人思考：

  选 parent_id 自关联而不是闭包表 / 路径枚举的原因：

  ┌──────────────┬─────────┬──────────────┬──────────────┐
  │ 方案          │ 查询效率 │ 写入效率     │ 复杂度        │
  ├──────────────┼─────────┼──────────────┼──────────────┤
  │ parent_id    │ 中      │ 高           │ 低            │
  │ 路径枚举      │ 高      │ 低(需重算)   │ 中            │
  │ 闭包表        │ 极高    │ 低(双表写)   │ 高            │
  │ 嵌套集        │ 高      │ 极低         │ 高            │
  └──────────────┴─────────┴──────────────┴──────────────┘

  本项目分类最多 2 层，parent_id 完全够用，没有必要引入额外表。
  但接口里仍然暴露了 getTreeList()，因为管理后台确实需要整棵树，
  这体现了"按需暴露能力"的设计哲学。


--------------------------------------------------------------------------------
5.6  软删除 vs 硬删除：status 状态机
--------------------------------------------------------------------------------

问题：
  用户"删除"一篇文章，是 DELETE FROM article WHERE ... 还是 UPDATE status=0？

结论：业务上所有"删除"都用 status 软删除，理由：

  1) 关联数据安全
     文章被删后，like_record / comment / footprint 仍要保留 user_id
     与 article_id 的关系用于回溯，硬删会破坏外链

  2) 数据可恢复
     用户误删时可通过 status=1 恢复，硬删不可逆

  3) 数据分析
     运营/数据团队需要历史内容做用户行为分析，软删保留全部痕迹

  4) 审计合规
     非遗内容平台涉及文化资产，删除操作需要可追溯

  status 字段在每张表里的语义：
    user.status            0 禁用  1 正常
    article.status         0 草稿  1 已发布  2 下架
    post.status            0 删除  1 正常
    comment.status         0 删除  1 正常
    course.status          0 下架  1 上架
    banner.status          0 禁用  1 启用

个人价值：
  软删除不是"懒得写 DELETE"，而是有意识的数据治理策略。
  当面试官问"为什么不用硬删除"时，这一套理由足以体现工程深度。


--------------------------------------------------------------------------------
5.7  MySQL 5.0 兼容性：删除全文索引，改为 LIKE 兜底
--------------------------------------------------------------------------------

  真实工程场景里，我曾经被问："为什么你的 article 表有 FULLTEXT 索引
  但实际查询里又用 LIKE？"答案是：

  1) schema_mysql5.sql 不能用 FULLTEXT（5.0/5.1 不支持 InnoDB 全文索引）
  2) MySQL 自带的全文索引不支持中文分词（5.6 起 InnoDB 支持但仍不
     适合中文），生产环境通常要接 Elasticsearch
  3) 教学项目里加 FULLTEXT 会让初学者以为"这就是搜索的全部"

  因此本项目主动放弃 FULLTEXT，把搜索功能下沉到：
    a) 业务层：title LIKE '%keyword%'（5 万级数据完全够用）
    b) 运维层：未来量级起来后再接入 ES

  这是"不教坏学生"的设计哲学：不留误导性的索引，
  也不留永远跑不通的 SQL。


--------------------------------------------------------------------------------
5.8  字符集与排序规则：utf8 / utf8mb4 的取舍
--------------------------------------------------------------------------------

  MySQL 的 utf8 是"假 utf8"，最多 3 字节，无法存储 emoji 表情。
  真正的 utf8 是 utf8mb4（4 字节）。

  本项目：
    5.5+ 版本用 utf8mb4（支持 emoji、𪚥 之类的生僻字）
    5.0/5.1 版本回退到 utf8（生僻字会变成 ?，但能跑）

  排序规则 utf8_unicode_ci：
    * _ci = case insensitive，不区分大小写
    * 比 utf8_general_ci 更准确，遵循 Unicode 排序标准
    * 性能略低但对中文业务更友好

  为什么不统一用 utf8mb4_general_ci？
    * 5.0 不支持 utf8mb4，强行统一会丢掉老版本用户
    * 排序规则的差异对绝大多数场景无感知


--------------------------------------------------------------------------------
5.9  密码加密：MD5 的历史选择与升级建议
--------------------------------------------------------------------------------

  当前实现（UserServiceImpl.encryptPassword）：
    return DigestUtils.md5DigestAsHex(password.getBytes(StandardCharsets.UTF_8));

  选择 MD5 的原因（坦白讲）：
    * 教学项目追求简单可读
    * MD5 + 简单字符串拼接在过去十几年是行业标准
    * 项目交付时演示友好

  已知风险：
    * 无盐值 MD5 可被彩虹表秒破
    * 现代 GPU 算力下 6 位密码秒解
    * Spring Security 强烈建议 BCrypt

  升级路径（生产环境必做）：
    private String encryptPassword(String password) {
        return BCrypt.hashpw(password, BCrypt.gensalt(12));
    }
    验证时：
    BCrypt.checkpw(rawPassword, user.getPassword())

  个人价值：
  在项目里不回避 MD5 的局限性，且写出明确的升级路径，
  这是诚实且专业的工程态度。比起"假装不知道"或"加个盐就觉得安全了"
  的做法，更能体现技术深度。


--------------------------------------------------------------------------------
5.10 时间字段：create_time / update_time 的统一规范
--------------------------------------------------------------------------------

  14 张表里所有时间相关字段都遵循：
    * create_time：业务记录首次落库时间
    * update_time：业务记录最后一次被修改时间（无则不设）

  字段类型：
    * DATETIME（不是 TIMESTAMP）
    * 范围 1000-01-01 ~ 9999-12-31，远超 TIMESTAMP 的 2038 年问题
    * 不受时区影响（TIMESTAMP 会被自动转换为 UTC）

  Java 端：
    使用 java.time.LocalDateTime 映射，对应 MyBatis-Plus 的
    MetaObjectHandler 自动填充策略（理论上可以，这里为简化由 Service
    手动 setCreateTime(LocalDateTime.now())）

  个人价值：
  14 张表的时间字段定义完全一致，这种"一致性"看似简单，
  实际是大型项目协作的基石。代码 review 时只需看字段名就能确定语义，
  极大降低沟通成本。

六、索引设计矩阵与查询性能思考
================================================================================

  ┌───────────────────┬──────────────────────────────────────────────┐
  │ 查询场景           │ 命中的索引                                    │
  ├───────────────────┼──────────────────────────────────────────────┤
  │ 用户登录           │ uk_user_phone / uk_user_email                 │
  │ 微信登录           │ uk_user_openid                                │
  │ 首页轮播           │ idx_banner_status, idx_banner_sort            │
  │ 首页热门           │ idx_article_hot + idx_article_status          │
  │ 分类页文章         │ idx_article_category + idx_article_create_time│
  │ 文章详情           │ PRIMARY(article_id)                           │
  │ 我的发布           │ idx_article_user + idx_article_create_time    │
  │ 动态列表           │ idx_post_create_time                          │
  │ 教程列表           │ idx_course_category + idx_course_difficulty   │
  │ 关注数             │ idx_follow_user_id                            │
  │ 粉丝数             │ idx_followed_user_id                          │
  │ 我的点赞           │ uk_like_user_target(覆盖)                     │
  │ 我的收藏           │ uk_collect_user_article                       │
  │ 我的足迹           │ idx_footprint_user + idx_footprint_time       │
  └───────────────────┴──────────────────────────────────────────────┘

  反例（未加索引、潜在慢查询）：
    * article.content LIKE '%关键字%'  → 全表扫（已规避）
    * 跨表 JOIN user 和 article 按 nickname 模糊查询
      → idx_user_nickname + idx_article_user 可缓解，但百万级仍慢
    * 未分表的 footprint 单用户 10w+ 条 → 需定时归档

  个人价值：
  主动梳理"哪些查询是热路径、哪些是冷路径"，
  决定索引资源向哪些字段倾斜，这是 DBA 视角与纯开发的本质区别。

七、事务边界与并发控制（结合 ServiceImpl 分析）
================================================================================

  本项目的事务粒度统一在 Service 层，标记方式：
    @Transactional(rollbackFor = Exception.class)

  关键事务场景：

  1) wxLogin（UserServiceImpl）
     事务范围：查询用户 + 首次登录插入 / 二次登录更新
     并发风险：两个请求同时发现用户不存在 → 双写
     应对：依赖 uk_user_openid 唯一约束，第二个事务直接抛异常
     反思：可以引入分布式锁或"先查后插"双重检查，简化版用唯一索引
           兜底已经足够

  2) 文章点赞 toggle（ArticleServiceImpl）
     事务范围：查 like_record → 存在删/不存在插 → update article.like_count
     并发风险：两次同时点赞，可能出现"两条 like_record"
     应对：UNIQUE (user_id, target_type, target_id) 唯一索引兜底
     反思：updateLikeCount 用 set like_count = like_count + 1
           的原子 SQL 是最佳实践，本项目 Mapper 层就是这种写法

  3) 关注切换 toggleFollow（FollowServiceImpl）
     事务范围：查 follow → 存在删/不存在插
     并发风险：同 2)
     应对：UNIQUE (user_id, follow_user_id)

  4) 学习时长累加（UserServiceImpl.updateStudyHours）
     事务范围：getById → 累加 → updateById
     并发风险：读后写丢失更新
     应对：理想方案是 WHERE study_hours = ? 的乐观锁，
           本项目为简化用"读+改+写"单事务串行化解决
     反思：用户学习时长是非敏感数据，最终一致即可，无需强一致

  个人价值：
  事务不是"加 @Transactional 就完事"，
  必须分析每段事务的并发场景，并选择"唯一索引/原子 SQL/乐观锁"等
  合适的兜底策略。本文档给出 4 个真实场景的反思，
  这是比代码本身更值钱的设计经验。

八、敏感数据与安全设计
================================================================================

  1) 密码
     * 存储：MD5(password)（生产环境应升级 BCrypt）
     * 传输：HTTPS（部署层兜底，代码不涉及）
     * 出参：UserController 显式从返回数据中剔除 password 字段

  2) 微信身份
     * openid / unionid 仅用于登录态，不在列表接口中返回
     * UserController.login() 手动封装 Map 过滤敏感字段
     * 这种"宁少勿多"的策略，比"全部返回再让前端过滤"更安全

  3) 接口幂等
     * 点赞/收藏/关注/下载都有 UNIQUE 索引保证
     * 重复请求会被识别为"已存在"，不会污染数据

  4) SQL 注入
     * MyBatis-Plus 默认使用 #{} 预编译参数
     * 业务 SQL 都走参数绑定，无 ${} 拼接

  5) 越权访问
     * 当前未做：未登录用户可访问大部分接口
     * 改进方向：基于 JWT + Spring Security 的认证拦截器
     * 建议优先保护：发布/删除/管理类接口

  个人价值：
  上述 5 条都是从代码里"挖"出来的安全设计点，
  写出来就是简历上"具备安全编码意识"的硬证据。

九、可扩展性预留
================================================================================

  在不破坏现有结构的前提下，本数据库已为以下扩展预留接口：

  1) 多种内容形态
     * like_record.target_type / comment.target_type 已支持任意扩展
     * 新增"作品点赞""课程评论"无需改表

  2) 课程订单化
     * 当前 course 是免费的"教程"
     * 未来若要收费，可新增 course_order 表，复用 user_id + course_id

  3) 积分 / 成就系统
     * user 表预留 study_hours 字段
     * 可新增 user_point / user_achievement 表，与 user 1:N

  4) 内容付费
     * article / course 都有 is_hot 运营位标记
     * 未来加 is_vip 字段即可，无须改表结构

  5) 多语言
     * 当前所有表都是中文
     * 国际化时建议新增 article_i18n 翻译表，不动主表

  个人价值：
  "演进式架构"是后端架构师的核心能力。
  本项目在 14 张表里留了 5 个清晰的扩展点，
  既不过度设计（没有提前建一堆空表），又足以支撑未来 1-2 年的迭代。
